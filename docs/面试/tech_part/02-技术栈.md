## 1. 技术栈全景

### 1.1 核心技术栈总览

| 层次 | 选用技术 | 关键版本/配置 | 选型理由 |
|------|---------|-------------|----------|
| **LLM Provider** | DeepSeek (OpenAI-compatible) | API Base, Chat Completion | 性价比高，支持长上下文，支持推理模型 |
| **Agent 编排** | LangGraph + LangChain Core | StateGraph, conditional edges | 可视化 DAG，支持 checkpoint，支持 conditional routing |
| **后端框架** | FastAPI + Uvicorn | async/await, Pydantic v2 | 高性能异步 API，自动 OpenAPI 文档，v2 模型校验强 |
| **持久化** | PostgreSQL + SQLAlchemy 2 | JSONB, async session | 全程 async，支持 JSONB 存储半结构化数据，迁移友好 |
| **向量检索（本地）** | FAISS | IndexFlatIP / IndexHNSW | 轻量、快速、支持内积相似度 |
| **搜索聚合** | SearXNG | 多引擎聚合 | 无需 API key，支持多源并行 |
| **前端** | React 19 + TypeScript 5 | 函数式组件 + hooks | 快速迭代，强类型，生态成熟 |
| **流式推送** | Server-Sent Events (SSE) | asyncio.Queue + EventEmitter | 轻量、无需 WebSocket 协议开销 |
| **评测** | pytest + FastAPI TestClient | Layer 1/2/3 分层 | 分层评测，Layer 1 无需 LLM API |
| **PDF 解析** | pdfplumber / arxiv API | — | 无需 OCR，学术场景 PDF 结构化 |

## 2. 各层技术选型详解

### 2.1 LLM 层 — DeepSeek (OpenAI-compatible API)

**代码位置**：`src/agent/llm.py`

```python
# 构建 reasoning LLM（支持推理模型如 DeepSeek Reasoner）
def build_reason_llm(
    settings,
    timeout_s: int | None = None,
    max_tokens: int | None = None,
    temperature: float | None = None,
) -> "ChatAnthropic":
    from langchain_anthropic import ChatAnthropic
    from langchain_openai import ChatOpenAI

    provider = settings.llm_provider or "openai"
    model = settings.reason_model

    if provider == "anthropic":
        llm = ChatAnthropic(
            model=model,
            anthropic_api_key=settings.anthropic_api_key,
            timeout=timeout_s or 60,
            max_tokens=max_tokens or 4096,
            temperature=temperature or 0.0,
        )
    else:  # openai-compatible（包括 DeepSeek）
        llm = ChatOpenAI(
            model=model,
            api_key=settings.openai_api_key,
            base_url=settings.openai_api_base,
            timeout=timeout_s or 60,
            max_tokens=max_tokens or 4096,
            temperature=temperature or 0.0,
        )
    return llm
```

**优势体现**：
- **OpenAI-compatible**：一套代码切换 OpenAI / DeepSeek / Qwen / 本地模型，无需改业务逻辑
- **多模型分离**：`reason_model`（复杂推理）vs `quick_model`（快速决策）分离，降低成本
- **超时控制**：`timeout_s` 参数精细控制每类调用的超时

**关键特性**：通过 `ChatOpenAI` + `base_url` 支持任意 OpenAI-compatible 端点，DeepSeek 的长上下文（128k tokens）和推理模型（DeepSeek-R1 风格）均可无缝接入。

### 2.2 Agent 编排 — LangGraph StateGraph

**代码位置**：`src/research/graph/builder.py`

```python
def build_research_graph(emitter: NodeEventEmitter | None = None):
    from langgraph.graph import END, StateGraph
    from src.research.graph.state import AgentState

    workflow = StateGraph(AgentState)

    # 注册 7 个节点
    workflow.add_node("clarify", instrument_node("clarify", run_clarify_node, emitter))
    workflow.add_node("search_plan", instrument_node("search_plan", run_search_plan_node, emitter))
    workflow.add_node("search", instrument_node("search", search_node, emitter))
    workflow.add_node("extract", instrument_node("extract", extract_node, emitter))
    workflow.add_node("draft", instrument_node("draft", draft_node, emitter))
    workflow.add_node("review", instrument_node("review", review_node, emitter))
    workflow.add_node("persist_artifacts", instrument_node("persist_artifacts", persist_artifacts_node, emitter))

    # 线性边
    workflow.add_edge("clarify", "search_plan")
    workflow.add_edge("search_plan", "search")
    workflow.add_edge("search", "extract")
    workflow.add_edge("extract", "draft")
    workflow.add_edge("draft", "review")

    # 条件边（clarify → END 或 search_plan）
    workflow.add_conditional_edges(
        "clarify",
        _route_after_clarify,
        {"search_plan": "search_plan", END: END},
    )

    # review → persist 或 END
    workflow.add_conditional_edges(
        "review",
        _route_after_review,
        {"persist_artifacts": "persist_artifacts", END: END},
    )

    workflow.set_entry_point("clarify")
    return workflow.compile()
```

**优势体现**：
- **DAG 可视化**：通过 `@xyflow/react` 将编译后的 graph 对象序列化给前端，实时展示节点状态
- **Conditional Edges**：clarify 失败 → END，review 通过 → persist，通过路由函数实现分支逻辑
- **Instrumentation 集成**：每个节点通过 `instrument_node` 包装，自动计时、发射事件、捕获异常
- **State 累积**：TypedDict 通过 `operator.add` 支持 tokens_used / warnings 的累加合并

### 2.3 后端框架 — FastAPI + Pydantic v2

**优势体现**：

1. **自动请求校验**：Pydantic v2 的 `model_validate` + `field_validator` 替代手动校验

```python
# src/api/routes/tasks.py
@router.post("/tasks/{task_id}/events")
async def task_events(request: Request, task_id: str):
    queue: asyncio.Queue = _get_event_queue(task_id)
    async def event_generator():
        while True:
            event = await queue.get()
            yield f"data: {json.dumps(event)}\n\n"
    return EventSourceResponse(event_generator())
```

3. **SSE 推送**：通过 `EventSourceResponse` 实现服务端推送，无需 WebSocket

### 2.4 持久化 — PostgreSQL + SQLAlchemy 2 + JSONB

**代码位置**：`src/db/models.py`, `src/db/task_persistence.py`

**优势体现**：

class TaskSnapshot(Base):
    __tablename__ = "task_snapshots"

## 3. 技术栈总结

### 3.1 架构风格：复合编排

```
Anthropic Workflow Patterns：
├── Prompt Chaining        ← LangGraph 线性边（clarify→search_plan→...→persist）
├── Evaluator-Optimizer   ← draft_node → review_node 循环
├── Parallelization        ← search_node 内 ThreadPoolExecutor 并行三源检索
└── Router/Orchestrator   ← Supervisor LEGACY/V2 切换 + conditional edges
```

### 3.2 选型原则

| 原则 | 体现 |
|------|------|
| **PostgreSQL-only** | 禁止 SQLite，所有持久化走 DATABASE_URL |
| **LLM 厂商无关** | OpenAI-compatible 接口，一套代码切换模型 |
| **Structured Output** | Pydantic v2 全链路约束，JSON Schema 驱动 LLM 输出 |
| **Observability-first** | instrument_node + NodeEventEmitter + trace_wrapper |
| **Async-first** | FastAPI async + SQLAlchemy async session |

## 4. 技术债务与改进方向

| 债务 | 现状 | 改进方向 |
|------|------|---------|
| **LLM 超时无统一重试** | `build_reason_llm` 无重试逻辑 | 集成 circuit_breaker |
| **向量检索未启用 pgvector** | 用 FAISS 本地替代 | PostgreSQL + pgvector 统一持久化 |
| **多模型未真正切换** | 始终用 DeepSeek | 支持 OpenAI / Anthropic / Qwen 动态切换 |
| **SSE 无心跳保活** | 长时间空闲可能断开 | 加心跳 ping / 重连逻辑 |
